## はじめに

ChatGPT や Claude のような AI エージェントが、人の代わりに買い物をする時代がすぐそこまで来ています。「米を探して買っておいて」とお願いしたら、エージェントが自分で店を見つけ、商品を理解し、決済まで済ませてくれる。そんな世界です。

ですが、ここで開発者として一度立ち止まって考えたいことがあります。**自分のサイトは、その AI エージェントから「見つけてもらえる」状態になっているでしょうか？**

この記事は、私が開発している JPYC EC Platform（JPYC ステーブルコインで決済できるマルチテナント EC）を、**AI ショッピングエージェントから発見・理解・購入されるサイト**に作り変えた記録です。エージェントが実際に商品ページへ辿り着き、決済まで完走できる状態。そこまでに何を実装したかを、実際のコードを引用しながらお伝えできたらと思います。
https://ec.jpyc-service.com/

「自分の EC や Web サービスを AI エージェント時代に対応させたい」という開発者の方を想定しています。結論を先に言うと、必要な仕事は4つの層に分かれていました。そして私は、そのうち**いちばん難しい層（決済）だけが完成していて、残り3層が空白**という、よくある「順番を間違えた」状態から出発しています。同じ轍を踏まないための順番も含めて、書いていきます！

## 1. なぜ今これをやるのか

### 1.1 「95% は来ているのに、20% しか読めない」

きっかけは、AI エージェントの買い物に関する一次情報を読み込んだことでした。PayPal の調査（米国 498 社対象の「Agentic Commerce Pulse Report」）が指摘していたのは、ざっくりこういう話です。**約 95% の事業者が AI エージェントのトラフィックを観測しているが、商品カタログを機械可読な構造化データにできている事業者は 20% ほどしかいない**、というものです。

https://www.paypal.com/us/brc/article/agentic-commerce-pulse-report-findings

人間の買い物客にとって「サイトが速い」「デザインが良い」は意味があります。ですが AI エージェントにとって意味があるのは別のことです。つまり、**構造化されているか、能力が宣言されているか、対話できるか、決済できるか**。この4つが揃っていないサイトは、エージェントから見ると「存在しないのと同じ」になってしまいます。

### 1.2 エージェンティック・コマースの4層スタック

業界で形成されつつある技術スタックは、下から積み上げると4層になります。

| 層 | 名称 | 役割 | 代表的な技術 |
|---|---|---|---|
| 第1層 | データ層（意味論） | 商品データを機械可読に | Schema.org / JSON-LD |
| 第2層 | 発見層（ディスカバリ） | 店舗の能力を宣言 | `.well-known` manifest、x402 Bazaar |
| 第3層 | 対話層（インタラクション） | 検索・カート構築・対話 | MCP（Model Context Protocol） |
| 第4層 | 取引層（決済） | 支払いの実行 | x402、ACP、AP2 |

この4層は独立ではなく、**下が上を支える**関係になっています。決済（第4層）がいくら立派でも、エージェントがその店に「辿り着け」なければ（第2層）、商品を「理解」できなければ（第1層）、決済画面まで到達しません。

ちなみに第4層に並べた x402 と AP2 はライバルではなく相互補完の関係です。**Google が 2025 年 9 月に発表した AP2（Agent Payments Protocol）には、ステーブルコイン決済レーンとして x402 が公式拡張として組み込まれています**。60 以上の組織（Coinbase・Mastercard・MetaMask・Ethereum Foundation など）が AP2 に参画していて、AI エージェント決済の主要プロトコルが揃って x402 を採用している状況です。詳細は §7 で改めて触れます。

<ここに4層スタック（第1層 意味論 → 第2層 発見 → 第3層 対話 → 第4層 取引）を下から積み上げたピラミッド図。各層に代表技術（Schema.org / .well-known / MCP / x402）を添え、「下が上を支える」関係を矢印で示す図を挿入>

### 1.3 出発点の診断

最初にやったのは、自分のプロジェクトがこの4層のどこにいるかを正直に測ることでした。コードを実地調査してみると、なかなか歪な状態でした。

最上層の第4層（取引層・x402）はほぼ完成している。ですがその下の第3層（対話層・MCP）と第2層（発見層・`.well-known`）は**まったくの空白**。第1層（意味論・Schema.org）も、トップページに少し書いてあるだけで、肝心の商品ページは手つかず。

逆説的ですが、**いちばん難しい第4層（決済プロトコル）を先に作り込んでいたために、その上に乗るべき1〜3層が抜け落ちていました**。立派なレジを作ったのに、店の看板も商品の値札も接客係も置いていない状態です。これではエージェントは店にすら辿り着けません。

この記事は、その空白を第1層から順に埋めていった記録です。なぜ第1層からなのかというと、投資対効果が層の番号が若いほど高いからです。データが既に DB にあるので第1層は数日で埋まり、効果も即時に出ます。下の層ほど「安く・早く・効く」のだから、下から始めるのが素直だと考えました。

## 2. 第1層：Schema.org / JSON-LD（最も低コストで最も重要）

それでは、いちばん下の第1層から埋めていきましょう。ここは「商品データを機械可読にする」層です。コストが最も低く、それでいて効果が即時に出る、いわば**いちばんおいしい層**だと思っています。

### 2.1 なぜ JSON-LD が効くのか

現代の LLM は **Schema.org のマークアップを極めてよく理解できるよう学習されています**。Schema.org は RSS の生みの親でもある R.V. Guha が設計に関わった語彙で、検索エンジンが20年使ってきた蓄積があります。だから「Schema.org を正しく書く」ことが、そのまま「LLM に読まれやすくする」最短経路になると思っています。

出発点では、トップページにだけ JSON-LD があり、**最も商業価値の高い商品ページが空白**でした。商品ページには Open Graph メタタグ（SNS シェア用）しかなく、`Product` も `Offer` も `AggregateRating` もありませんでした。

### 2.2 実装：production truth から生成する

実装で絶対に守るべき原則がひとつあります。それは、**JSON-LD の価格・在庫は、表示されている実データと完全に一致させる**ことです。「在庫切れなのに InStock」といった食い違いは Google も LLM も検知し、信頼度評価を下げてしまいます。だから JSON-LD は、価格や在庫を持っているシステム（DB）から直接生成すべきで、CMS プラグインで「とりあえず付けた」レベルだと逆効果になりうると考えています。

私は JSON-LD を組み立てる純粋関数として用意しました。引数ですべての値を受け取るので、サーバーコンポーネントが DB の生の値を渡します。そのため、キャッシュ越しの古い値が混入する余地がありません。

```typescript
/**
 * Build a schema.org Product node (with nested Offer / AggregateOffer and an
 * optional AggregateRating) for a single product page.
 */
export function buildProductJsonLd(input: ProductJsonLdInput): JsonLdObject {
  const { siteUrl, productUrl, product, shop, rating } = input

  const seller: JsonLdObject = {
    "@type": "Organization",
    name: shop.name,
    url: `${siteUrl}/shops/${shop.slug}`,
  }

  const node: JsonLdObject = {
    "@context": "https://schema.org/",
    "@type": "Product",
    name: product.name,
    url: productUrl,
    offers: buildOffers(product, productUrl, seller),
  }

  if (product.description) node.description = product.description
  if (product.image_urls.length > 0) node.image = product.image_urls
  if (product.sku) node.sku = product.sku
  if (product.category) node.category = product.category
  node.brand = { "@type": "Brand", name: shop.name }

  // Explicit JPYC amount alongside the JPY-denominated Offer price.
  node.additionalProperty = {
    "@type": "PropertyValue",
    name: "JPYC建て価格",
    value: product.price_jpyc,
    unitText: "JPYC",
  }

  const aggregate = buildAggregateRating(rating)
  if (aggregate) node.aggregateRating = aggregate

  return node
}
```

ここには、汎用 EC とは違う JPYC ならではの判断が2つ入っています。

**(1) 価格通貨は JPY、JPYC 額は `additionalProperty` で併記する。** Schema.org に「JPYC」という通貨コードはありません。`priceCurrency` には `JPY` を入れて「これは日本円建ての数字だ」とエージェントに伝えつつ、別途 `additionalProperty` に JPYC 建ての明示額を載せます。これでエージェントは「日本円として扱える」と「この店はステーブルコインで決済できる」の両方を知ることができます。

**(2) 暗号決済の宣言は `acceptedPaymentMethod`。** Schema.org には `CryptoCurrency` が定義されています。`Offer` にこれを入れることが、エージェント側で「この店は暗号決済できる」と判定する根拠になります。

```typescript
// buildOffers() の一部。Offer に必ず入れる共通フィールド
const common: JsonLdObject = {
  priceCurrency: "JPY",
  availability,                                       // InStock / OutOfStock
  itemCondition: "https://schema.org/NewCondition",
  acceptedPaymentMethod: "https://schema.org/CryptoCurrency",
  url: productUrl,
  seller,
}
```

バリエーション商品（サイズ・色）は単一価格を持たないので、`Offer` ではなく `AggregateOffer`（`lowPrice` / `highPrice`）を出します。レビューが0件の商品では `aggregateRating` を**出しません**。空の rating は逆にリッチリザルトの評価を下げてしまうからです。

### 2.3 サーバーサイドで `<head>` に埋め込む

ここが落とし穴になりやすいところです。**クライアント JS で後から JSON-LD を差し込むと、LLM クローラーに読まれないことがあります。** Next.js の App Router ならサーバーコンポーネントで生成するのが定石です。私は商品ページのサーバーコンポーネントで、レビュー集計を取ってから JSON-LD を埋め込みました。

```typescript
// schema.org Product JSON-LD — embedded server-side so LLM/search crawlers
// and shopping agents read live price / stock / rating from the page head.
const rating = await reviewRepository.getAverageRating(product.id)
const productJsonLd = buildProductJsonLd({
  siteUrl,
  productUrl: `${siteUrl}/shops/${shop.slug}/products/${product.id}`,
  product: { /* DB の生の値 */ },
  shop: { name: shop.name, slug: shop.slug },
  rating,
})

return (
  <>
    <script
      type="application/ld+json"
      dangerouslySetInnerHTML={{ __html: JSON.stringify(productJsonLd) }}
    />
    {/* ... */}
```

### 2.4 JSON-LD だけでは足りない。可視テーブルも置く

意外な一次情報がありました。Evil Martians の記事「Making your site visible to LLMs」では、**「商品データを JSON-LD だけに入れると LLM に読まれず、効いたのはどれも可視テキストを充実させる施策だった」**という実験結果が紹介されています。

https://evilmartians.com/chronicles/how-to-make-your-website-visible-to-llms

これは「JSON-LD が無意味」という意味ではありません。**JSON-LD を書いた上で、商品ページ本文にも仕様表を可視のテーブルとして置く**。つまり、両方やるのが正解、ということだと思います。私は散在していた商品属性（価格・決済方法・カテゴリ・SKU・バリエーション・在庫・配送・対応チェーン）を、ひとつの `<table>` にまとめました。`<th scope="row">` で各行にラベルを付け、機械にも人にも読める構造にしてあります。

```tsx
function Row({ label, children }: { label: string; children: React.ReactNode }) {
  return (
    <tr className="border-b border-theme-border last:border-b-0">
      <th scope="row" className="...">{label}</th>
      <td className="...">{children}</td>
    </tr>
  )
}
```

ショップページにも同様に `OnlineStore` の JSON-LD を埋め込みました。店名・ストーリー・ロゴ・JPYC 決済対応、そして SNS リンクを `sameAs` に入れます。`sameAs` はエージェントが「この出店者は誰か」を外部プロファイルと突き合わせる手がかりになります。

**第1層の結果**：商品ページとショップページが、live な価格・在庫・評価を構造化データとして出すようになりました。コストは2日、効果は即時です。

## 3. 第2層：`.well-known` Manifest（エージェントへの「名刺」）

### 3.1 商品の意味と、店舗の能力

第1層が「個々の商品の意味」を伝えるのに対し、第2層が伝えるのは「**店舗全体としてどんな能力を持ち、どのエンドポイントを使えばエージェントと取引できるか**」です。これは店の「名刺」にあたります。

エージェント向け manifest のパスは複数並立しています。Google UCP の `/.well-known/ucp/manifest.json`、Google A2A の `/.well-known/agent-card.json`、Open Agentic Commerce の `/.well-known/commerce-manifest`、旧 OpenAI Plugin の `/.well-known/ai-plugin.json` などです。クリプトネイティブの中小プラットフォームには、決済レイヤー非依存でベンダー中立な **Open Agentic Commerce Manifest** がいちばん適合すると考えています。

私は manifest 群を**ひとつの真実の源**（純粋関数）から生成しました。3つの `.well-known` ドキュメントが互いに食い違わないようにするためです。

```typescript
// payment rails は環境に自動追従する
function buildPaymentRails(facilitatorUrl: string): JsonObject[] {
  return getAvailableChains().map((chain) => ({
    type: "x402",
    scheme: "exact",
    network: `eip155:${chain.id}`,
    chainName: chain.name,
    asset: chain.jpycContractAddress,
    assetSymbol: "JPYC",
    assetDecimals: 18,
    facilitator: facilitatorUrl,
  }))
}
```

`getAvailableChains()` は環境（本番＝メインネット / staging＝テストネット）で返すチェーンが変わります。だから manifest の決済レール一覧は**手で書き換えなくても環境に追従します**。本番では Ethereum/Polygon/Avalanche、staging では Sepolia/Amoy/Fuji が自動で並びます。

### 3.2 Next.js の落とし穴：ドット始まりディレクトリ

ここで**実装が一度まるごと無効になっていた罠**を踏みました。

`.well-known` ルートを Next.js App Router で素直に作ると、`app/.well-known/commerce-manifest/route.ts` と書きたくなります。型チェックも通り、開発中は動いているように見えます。

ところが**クリーンビルドのルート一覧に `.well-known` が出てきませんでした**。原因は **Next.js はドット（`.`）始まりのディレクトリを「プライベートフォルダ」とみなし、ルーティングから除外する**ことでした。`app/.well-known/` も `app/llms.txt/` も、ドット始まりなので静かにルート化されていなかったのです。第2層で作った4ルートが、実は全部 404 だったわけです。

これは「ビルドは exit 0、型チェックも通る、でも実は配信されていない」という、いちばん気づきにくい壊れ方だと思います。気づけたのは、**コミット前に必ずクリーンビルドしてルート一覧を目視確認する**習慣のおかげでした。

修正は Next.js 公式の解決策に沿うものでした。ディレクトリを通常名（`app/well-known/`、`app/llms-txt/`）にして、`next.config.ts` の `rewrites` で公開パスにマップしました。

```typescript
// next.config.ts
async rewrites() {
  return [
    // Next.js treats dot-prefixed app/ directories as private folders and
    // won't route them, so the agent-discovery routes live under plain
    // names (app/well-known, app/llms-txt) and are surfaced at their
    // conventional public paths via these rewrites.
    { source: '/.well-known/:path*', destination: '/well-known/:path*' },
    { source: '/llms.txt', destination: '/llms-txt' },
  ]
}
```

修正後、ビルドのルート表に `/well-known/commerce-manifest`、`/well-known/agent-card.json`、`/well-known/ai-plugin.json`、`/llms-txt` がすべて現れ、外部からは慣習どおりの `/.well-known/...` でアクセスできるようになりました。

**教訓**：エージェント向けのエンドポイントは「人間が普段アクセスしない URL」なので、壊れていても気づきにくいです。実 HTTP で叩いて確認するまでは「実装した」と言ってはいけないと思っています。

### 3.3 llms.txt は「やって損はない」程度

`llms.txt`（LLM 向けの Markdown インデックス）も用意しました。ただし正直に書くと、**主要 LLM クローラーが実際に llms.txt を参照している証拠は乏しい**ことが、複数のサーバーログ解析で示されています。Google も当初否定したあとに静かに自社版を出すなど、普及度・実効性とも未確定です。

私の判断は「ゼロコストだしやって損はないが、優先度は Schema.org や MCP よりずっと下」というものでした。実際 `/llms.txt` は sitemap と同じデータソースを使い回す軽い実装に留めています。

### 3.4 OpenAPI を「ファイル」から「エンドポイント」に

第2層の整備中に、もうひとつ古い問題が見つかりました。OpenAPI 仕様書は存在したのですが、次の2点で問題がありました。

- `info.description` が**旧フロー（EIP-3009 `receiveWithAuthorization`）のまま**で、現行の x402 `transferWithAuthorization` と矛盾していた
- **HTTP で配信するルートが無く**、外部エージェントが取得できなかった

「ドキュメントはあるが、エージェントには届かない」状態でした。仕様書を現行フローに書き直し、`/api/v1/openapi.yaml` で配信するようにしました。同期ズレを防ぐため、正本は1ファイルにまとめ、ビルド時に公開ディレクトリへコピーする（コピーは gitignore、`dev`/`build` で再生成）スクリプトを入れました。

## 4. 第3層：MCP サーバー（最大の差別化要因）

### 4.1 なぜ MCP が戦略の核なのか

第1層は「静的な商品データ」、第2層は「店舗の能力宣言」を伝えます。ですが**エージェントが実際に在庫を確認したり、「米の中で予算3000円以下で減農薬のものを探して」のような自然言語クエリを実行したりするには、対話的な API が必要**です。それを担うのが MCP（Model Context Protocol）サーバーです。

Shopify は2025年から複数の MCP サーバーを立て、ChatGPT・Claude・Gemini から直接ショップを検索・カート構築・チェックアウトできるようにしています。クリプト EC では Cryptorefills が「**MCP（対話）+ Agent Skills（能力公開）+ x402（決済）**」の3点セットを最先端の構成として展開しています。

### 4.2 設計判断1：ステートレス Streamable HTTP

MCP サーバーを Next.js の Route Handler として `POST /mcp` に立てました。ここで最初の重要な設計判断をしました。**ステートレスモード**にする、というものです。

MCP の Streamable HTTP transport はセッションを保持できますが、私はセッション ID を発行しません（`sessionIdGenerator: undefined`）。理由は、storefront が水平スケールするからです。複数インスタンスで動くと、MCP セッションを特定プロセスに固定できません。リクエストごとに `McpServer` と transport を新規に生成し、処理が終わったら破棄します。

```typescript
export async function POST(request: Request): Promise<Response> {
  const server = buildEcMcpServer()
  // Stateless: no session id, no cross-request state.
  const transport = new WebStandardStreamableHTTPServerTransport({
    sessionIdGenerator: undefined,
  })

  try {
    await server.connect(transport)
    const response = await transport.handleRequest(request)

    // Buffer the body before tearing the transport down — closing it while
    // the response stream is still open would truncate the reply.
    const body = await response.arrayBuffer()
    const headers = new Headers(response.headers)
    for (const [k, v] of Object.entries(CORS_HEADERS)) headers.set(k, v)

    return new Response(body, { status: response.status, headers })
  } finally {
    await transport.close().catch(() => {})
    await server.close().catch(() => {})
  }
}
```

実装上の小さな罠：`finally` で transport を閉じる前に、レスポンス body を `arrayBuffer()` で**読み切っておく**ことです。閉じてからストリームを返すと、レスポンスが途中で切れてしまいます。ステートレスな MCP ツール呼び出しは小さな JSON-RPC ペイロード1つなので、読み切るコストは無視できます。

なお SDK には `WebStandardStreamableHTTPServerTransport` という、Web 標準の `Request`/`Response` をそのまま受け取る transport がありました。これのおかげで Node の `req`/`res` への変換が一切不要になり、Next.js Route Handler にそのまま嵌まりました。

### 4.3 設計判断2：ツールの説明文にドメイン知識を埋める

x402 Bazaar のドキュメントが明確に書いているとおり、**ツールやパラメータの説明文（description）の質が、そのままエージェントの呼び出し精度に直結します**。「商品を検索します」では足りません。

私は MCP の読み取りツール（`list_shops` / `get_shop` / `list_products_in_shop` / `get_product` / `search_products`）の description に、JPYC EC らしいドメイン文脈を厚く埋め込みました。

```typescript
// search_products の description（抜粋）

  "search_products",
  {
    title: "商品検索",
    description:
      "JPYC EC Platform 上の公開商品を横断検索する。減農薬米・地酒・" +
```

### 4.4 設計判断3：公開 HTTP MCP は秘密鍵を持たない

第3層でいちばん重要な設計判断がこれです。

EC 本体の MCP は**不特定多数のエージェントが叩く公開エンドポイント**です。だから **買い手の秘密鍵を絶対に預かってはいけません。** 署名はエージェント側（クライアント）が自分のウォレットで行うべきだと考えています。

x402 は2往復の決済（後述）なので、取引ツールをこう設計しました。

- `quote_checkout`：x402 の1回目。在庫を5分間仮押さえし、金額サマリと**署名対象のチャレンジ**を返します。エージェントはこれを受け取ります
- （エージェントが自分のウォレットで EIP-3009 署名します。ここに MCP は関与しません）
- `submit_payment`：x402 の2回目。エージェントが**自分で署名した**ペイロードを渡してきます。MCP はそれを facilitator に渡して決済を確定します
- `get_order_status`：注文履歴

秘密鍵を持つのは、ローカルで動く別の MCP（`jpyc-skill` の stdio 版、エージェント所有の鍵を使う）です。**公開 HTTP MCP とローカル stdio MCP で役割を分ける**。これが正しい設計だと考えています。

### 4.5 MCP UI：テキストの壁を超える

参考にした一次情報で「JPYC EC の最大の差別化要因になりうる」と評されたのが、MCP UI です。MCP レスポンスに `ui://` リソース（HTML）を埋め込むと、対応ホスト（Claude / ChatGPT / Goose / VS Code）が**インタラクティブな商品カード**として描画してくれます。MCP UI のパターンは 2026年初頭に MCP Apps として公式標準化されました。

JPYC EC が出店者として推しているのは、減農薬米の生産者や伝統工芸の職人です。**作り手のストーリーや風景写真が、買う動機の大半を占めます。** テキストの JSON 羅列ではそれが伝わりません。だから `search_products` と `get_product` は、JSON に加えて `ui://` の HTML カード（画像・JPYC 価格・在庫・生産者のストーリー）を返すようにしました。

新しいパッケージは足さず、MCP 標準の `EmbeddedResource`（`type: "resource"` の content ブロック）を直接組み立てました。HTML はサンドボックス iframe で描画されるので、**ショップが書いたテキストや画像 URL は全て HTML エスケープ必須**です。`<script>` 注入を弾くテストも書きました。

## 5. 第4層：取引層を「動く」から「本番で壊れない」へ

x402（HTTP 402 "Payment Required" を復活させたステーブルコイン決済プロトコル）の実装自体は既にありました。ですが §1.3 で「ほぼ完成」と書いた第4層は、実は手放しで完成とは言えない状態でした。具体的には、**決済ロジックは動くものの、本番でスケールすると壊れる土台**の上に乗っていたのです。この章では、その「動くけれど本番では壊れる」部分を直していった話を書いていきます。

### 5.1 x402 の2往復フロー

まず x402 のチェックアウトは2回の HTTP リクエストで成立します。

1. **1回目**（署名なし）：客が「これ買う」と POST します。サーバが在庫を仮押さえし、注文内容のスナップショットを保存します。`reservation_id` を発行して HTTP 402 で返します
2. **2回目**（署名付き）：客が EIP-3009 署名と `reservation_id` だけを POST します。サーバはその ID で1回目のスナップショットを引き当て、署名と照合して決済します

つまり `reservation` は「1回目と2回目をつなぐ一時記憶」です。**2回目で必ず引き当てられないと決済が成立しません。**

<ここに x402 の2往復フロー図。1回目（客→サーバ：署名なし POST、サーバが在庫を仮押さえ→ reservation_id を発行して 402 を返す）と2回目（客→サーバ：EIP-3009 署名 + reservation_id を POST →スナップショットを引き当てて決済）を、reservation を中心に時系列で示す図を挿入>

### 5.2 in-memory Map という時限爆弾

問題は、この一時記憶が**サーバープロセスのメモリ上の `Map`** だったことです。メモリはそのプロセスの中にしか存在しません。これが2つの形で壊れます。

- **複数プロセスで動かす場合**：1回目がインスタンス A に、2回目が B に着くと、B のメモリに reservation がありません。`reservation_not_found` で**決済失敗**になります
- **再デプロイ・再起動の場合**：メモリが消え、進行中の reservation が全滅します

「1回目と2回目で同じプロセスに当たる」前提に賭けている状態です。低トラフィックなら今は動きますが、スケールした瞬間に**確率的に決済が落ちます**。しかも「たまに失敗する」ので原因が掴みにくい、最悪のバグの出方をします。AI エージェントは人間より桁違いに多くの並行リクエストを投げてくるので、エージェンティック・コマースの本番運用では特に致命的だと思っています。

<ここに in-memory Map が壊れる仕組みの図。複数プロセス（インスタンス A / B）があり、1回目のリクエストが A のメモリに reservation を保存、2回目が B に振られて「B のメモリに無い → reservation_not_found」になる様子を示す図を挿入>

修正は、reservation を Postgres テーブルに永続化することでした。

```sql
CREATE TABLE IF NOT EXISTS x402_reservations (
  id TEXT PRIMARY KEY,
  kind TEXT NOT NULL,
  snapshot JSONB NOT NULL,
  expires_at TIMESTAMPTZ NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

スナップショットは `JSONB` に**そのまま丸ごと**保存します。カートの構造は深い（`items[]` / gift / checkout_options / amounts）ので、平坦化せず1つの JSON 値として往復させることで、「**署名時に客が承諾した内容**」と「**DB に書かれる内容**」が証明可能に一致します。これは「DB 中心設計」であるこのプロジェクトに一貫した解でもありました。Redis を新規導入するより、既にある PostgreSQL を使うほうがインフラも増えません。

### 5.3 facilitator 呼び出しにリトライを足す

x402 の決済を実際にチェーン上で実行するのは facilitator（別リポの Cloudflare Workers / Node サービス）です。EC はそこに HTTP で verify / settle を投げます。ですが、facilitator が一瞬 503 を返したり、ネットワークが切れたりすると、本来成功できた checkout が落ちていました。リトライが無かったのです。

ここで「リトライしてよいか」を考える必要がありました。`/verify` は read-only なので安全です。ですが `/settle` は **on-chain 送金という副作用**があります。素朴にリトライすれば二重送金の危険があります。

調べてみると、facilitator の OpenAPI 仕様にこう書いてありました。**「同じ `(network, payer, nonce)` の組で settle 済みなら、再ブロードキャストせずキャッシュした結果を返す」**。facilitator が冪等に作られているのです。さらに EIP-3009 の nonce は使い切りで、`x402_payments` テーブルにも `UNIQUE(chain_id, payer, nonce)` 制約があります。**多層の防御で二重送金はプロトコルレベルで防がれています。** だから settle も安全にリトライできます。

```typescript
/**
 * Both /verify and /settle are safe to retry: /verify is read-only, and
 * the facilitator settles idempotently on `(network, payer, nonce)` — a
 * repeated /settle for an already-settled authorization returns the
 * cached result without re-broadcasting.
 *
 * Retryable: network/DNS failure, request timeout, HTTP 5xx, HTTP 429.
 * Not retryable: HTTP 4xx (a client error — retrying can't fix it).
 */
private async post(path: string, body: unknown): Promise<unknown> {
  const payload = JSON.stringify(body)
  let lastError: FacilitatorError | undefined

  for (let attempt = 1; attempt <= this.maxAttempts; attempt++) {
    try {
      return await this.postOnce(path, payload)
    } catch (e) {
      if (!(e instanceof FacilitatorError) || !e.retryable) {
        throw e
      }
      lastError = e
      if (attempt < this.maxAttempts) {
        // Exponential backoff: base, 2×base, 4×base, …
        await this.sleepFn(this.retryBaseMs * 2 ** (attempt - 1))
      }
    }
  }
  throw lastError ?? new FacilitatorError(/* ... */)
}
```

リトライ可否を `FacilitatorError.retryable` フラグで表現し、5xx・429・ネットワーク障害は指数バックオフでリトライ、4xx は即座に諦めるようにしました。

### 5.4 x402 Bazaar：決済層のディスカバリ

第2層は「サイトの発見」ですが、x402 には決済層独自のディスカバリ機構**x402 Bazaar** があります。これは「x402 で支払えるリソースのカタログ」で、エージェントがカタログを叩いて「払える API」を探せます。

CDP（Coinbase）の facilitator を使う場合は自動でカタログ化されますが、JPYC EC は**自前の facilitator** を運用しています。自前 facilitator が `GET /discovery/resources` を提供すれば、それが Bazaar 的なディスカバリを担えます。私は facilitator リポにこのエンドポイントを実装しました。

```typescript
// x402 Bazaar discovery layer — lets agents and clients enumerate the
// x402-payable resources this facilitator fronts. Catalog is static
// (config-driven) since the facilitator is DB-free.
app.get("/discovery/resources", (c) => {
  const all = deps.discovery?.resources ?? []
  const limit = /* clamp(query.limit, 1, 1000) */
  const offset = /* max(query.offset, 0) */
  const body: DiscoveryResourcesResponse = {
    x402Version: X402_VERSION,
    items: all.slice(offset, offset + limit),
    pagination: { limit, offset, total: all.length },
  }
  return c.json(body)
})
```

レスポンスの形は CDP Bazaar の `/discovery/resources` に合わせました。facilitator は DB-free 設計なので、カタログは「観測した settle から動的に作る」のではなく、環境変数（`X402_DISCOVERY_RESOURCES`、JSON 文字列）で**静的に宣言する**方式にしました。EC 側の commerce-manifest は、この discovery エンドポイントを指すようにしました。

### 5.5 カタログを「実際に設定する」で見えたもの

エンドポイントを実装しても、`X402_DISCOVERY_RESOURCES` が空なら `/discovery/resources` は空配列を返します。discovery が実稼働するのは、カタログ JSON を実際に設定したときです。その JSON を作る段で、マルチリポ構成ならではのことが3つ見えてきました。

**(1) 「全チェーン」は EC の全チェーンではない。** カタログには「x402 で払えるチェーン」を `accepts[]` として並べます。当然「対応チェーン全部」と考えてしまいますが、`accepts[]` に載せてよいのは **facilitator が settle できるチェーンだけ**です。facilitator の chain registry を見ると 8 チェーン（mainnet 4 + testnet 4）でした。一方 EC 側の registry には Arc というチェーンもありました。EC にあって facilitator に無いチェーンをカタログに載せると、エージェントは「払える」と思って 402 を受け取り、settle で facilitator に弾かれてしまいます。**カタログの真実の源は EC ではなく facilitator 側の registry** です。マルチリポでは「どちらが正か」を取り違えやすいと感じました。

**(2) 環境ごとに別のカタログ。** ステージング facilitator が有効化しているのはテストネット 4 チェーン、本番はメインネット 4 チェーンです。第2層で manifest を環境追従させた（§3.1）のと同じで、discovery カタログもステージング用と本番用を別に持ちます。`resource` の URL も `stg-ec.jpyc-service.com` / `ec.jpyc-service.com` で当然変わります。

**(3) カタログ JSON を、facilitator 自身のスキーマで事前検証する。** `X402_DISCOVERY_RESOURCES` は手で書く JSON です。間違えれば facilitator が起動時に弾きます（そして discovery が黙って無効になります）。設定してから気づくのは遅いです。facilitator のカタログパーサは `discoveryResourceSchema`（zod）で検証しているので、**設定する前に同じスキーマに通して**おきました。

```js
import { parseDiscoveryConfig } from "@jpyc-x402/facilitator"
// 設定予定の JSON 文字列を、facilitator が使うのと同じ検証に通す
const result = parseDiscoveryConfig(catalogJsonString)
//  → null なら facilitator も弾く。設定前に分かる
```

これは §6.3 で書く「機械可読であると同時に機械検証可能であれ」を、設定値そのものに適用しただけです。エージェント向けに JSON を出す側も、JSON を設定値として受け取る側も、同じスキーマを共有していれば、「本番に入れてから壊れていた」を一段手前で潰せます。

## 6. 道中で踏んだ「実装以外」の罠

4層を埋める作業で、コードそのものより厄介な問題に何度かぶつかりました。記録しておきます。

### 6.1 コードと矛盾する「嘘コメント」

調査の最初、別のサブエージェントに現状を調べさせたところ「x402 の settle は未実装、501 を返す」と報告してきました。ですが実際のコードを読むと、settle は完全に実装されていました。

原因は、checkout のルートハンドラのファイル冒頭コメントにありました。

```
*     → 200 OK (settle implementation: follow-up task; currently 501)
* ... it returns 501 with a clear marker so monitoring
```

このコメントは「これから実装する」前提の下書きで、実装が完了したのに**コメントだけ更新されずに残っていました**。`git log` で確認すると、コメントも実装も同じコミットで生まれていました。つまり、コメントが嘘をついていたのです。

人間のレビュアーも冒頭コメントを信じます。LLM エージェントは特に「コメント＞コード」と誤って重み付けしがちです。将来 MCP やエージェントがこのコードを読めば、同じ嘘を引き継いでしまいます。コメントを実装に合わせて修正し、`CLAUDE.md`（プロジェクト規約）に再発防止ルールを足しました。

> ファイルヘッダや関数の doc コメントが、その下のコードと矛盾する状態に
> してはならない。"not yet implemented" や "returns 501" のような古い
> コメントは、人間にも AI エージェント / MCP クライアントにも誤読される。
> 実装を終えたら、同じ変更で doc コメントも更新すること。

**エージェンティック・コマースの時代、コードコメントはエージェントが読む一次情報になります。** 嘘コメントは技術的負債を超えて、エージェントを誤動作させるバグになると思っています。

### 6.2 「ビルドが通る」と「配信されている」は違う

第2層の `.well-known` ルーティング問題（§3.2）が典型ですが、**型チェックが通り、ビルドが exit 0 でも、ルートが実際には配信されていない**ことがあります。Next.js のドット始まりディレクトリ問題はその一例でした。

教訓は単純です。**エージェント向けエンドポイントは、本番モードで起動して実際に HTTP で叩くまで「完成」と言わない。** 私は最後に `next start` で本番ビルドを起動し、全層を curl で検証しました。

```bash
# 第2層
curl -s -o /dev/null -w "%{http_code}\n" http://localhost:3000/.well-known/commerce-manifest
# → 200

# 第3層（MCP）
curl -s -X POST http://localhost:3000/mcp \
  -H "Content-Type: application/json" \
  -H "Accept: application/json, text/event-stream" \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/list","params":{}}'
# → 8 ツールが inputSchema 付きで返る

# tools/call も実 DB からショップを取得できた
```

### 6.3 構造化データは「機械可読」であると同時に「機械検証可能」であれ

第1層の JSON-LD も第2層の manifest も第4層の discovery レスポンスも、**zod スキーマで型を定義**しました。これは TypeScript の型チェックを通すためだけではありません。出力する JSON が「エージェントが期待する形」であることを、コードレベルで保証するためです。

たとえば facilitator の discovery レスポンスは、CDP Bazaar の形に合わせた zod スキーマで定義し、env から読むカタログもそのスキーマで検証しています。**不正な設定は起動時に弾かれ、空カタログにフォールバックします**。facilitator 本体（verify/settle）は決して巻き添えにしません。

## 7. 将来層：AP2（Agent Payments Protocol）

4層スタックの第4層には x402 以外の選択肢もあります。中でも **AP2**（Google 主導、60社以上）は「Intent / Cart / Payment という暗号署名された Verifiable Digital Credential でエージェント決済を認可する」プロトコルで、**x402 を決済レールとして使う公式統合（A2A x402 extension）**を持っています。

私は AP2 については**設計ドキュメントだけ**を書きました。理由は2つあります。

1. AP2 仕様は v0.2.0（2026年4月）時点でなお流動的で、マンデートが「3種」とも「2種×2ステージ」とも記述が揺れています
2. JPYC EC は既に x402 を完備しているので、AP2 対応とは「x402 をゼロから作り直す」ことではなく「**x402 の手前にマンデート検証層を被せる**」ことになります。仕様確定を待つ価値があると考えました

実質的な価値ははっきりしています。今 MCP の取引ツールにある `max_amount_atomic`（予算上限）は「エージェントが自主的に守る」もので、ハイジャックされたエージェントは無視できてしまいます。AP2 の Payment Mandate は**ユーザー署名で予算を縛ります**。ここが本物の安全性向上になります。仕様が安定したら実装したいと思っています。

## 8. まとめ：順番と、検証

今回は、AI エージェントから「見つけてもらえる」EC をつくるために、エージェンティック・コマースの4層スタックを第1層から順に埋めていった話を書いていきました。出発点では第4層（決済）だけが完成していて残り3層が空白でしたが、最終的には3つのリポジトリ（EC 本体 / x402 facilitator / agent skill）にまたがって全層を実装し、本番ビルドでの E2E 検証まで通すことができました。

振り返って、技術的に重要だった判断を3つ挙げてみます。

1. **第1層から始めた**：データが DB に揃っているので最も低コストで、かつ下の層は上の層の前提になります。順番が投資対効果を決めると思います
2. **公開 HTTP の MCP は秘密鍵を持たない**：鍵を持つローカル stdio MCP と役割を分けました。「不特定多数が叩くエンドポイント」の設計原則です
3. **スケールを前提に作り直した**：reservation の Postgres 化、MCP のステートレス化、facilitator のリトライ。AI エージェントは人間より並行リクエストが桁違いに多く、「たまに動く」では本番に出せません

そして、技術以前に効いた習慣がひとつあります。それは、**エージェント向けの成果物は、実際に HTTP で叩いて、実データが返るまで「完成」と呼ばない**ということです。型チェックもビルドも通るのにルートが 404、コメントは実装と矛盾、in-memory ストアはスケールで壊れる。これらは全部「動いているように見えて動いていない」類のバグでした。AI エージェントは人間の何倍も正直にサイトを叩いてきます。曖昧さはそのまま顕在化してしまいます。

「AI に見つけてもらえる EC」とは、結局のところ **構造化され、能力を宣言し、対話でき、決済でき、そしてそのすべてが実際に検証されている EC** のことだったのだと思います。

そう遠くない未来、人ではなく AI エージェントが私たちの代わりに買い物をする世界がやってきます。そのとき、生産者が丹精込めて作った減農薬米や伝統工芸品が、ちゃんとエージェントに「見つけてもらえる」かどうかは、今こうして4層を埋めておけるかにかかっていると思っています。技術が、作り手とその先の買い手をきちんと繋ぐ。そんな EC を、これからもつくっていきたいです！
