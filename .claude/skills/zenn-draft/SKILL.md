# Zenn記事下書き生成

既存記事から著者の文体を学習し、ネタ・解釈をもとに下書きを `_inbox/drafts/YYYYMMDD-{slug}.md` に生成する。

## 実行手順

### 1. 著者の文体・温度感を学習

`articles/` 内の全記事を読み込み、以下の特徴を把握する：

- **構成パターン**: 「はじめに」から始まる、章立て（## / ###）の粒度
- **トーン**: 親しみやすい口語体（「〜です！」「〜ですよね」）、難しい概念を例えで説明する
- **読者への配慮**: 「〜な方はここへ飛んでください」などの導線
- **自分の意見の入れ方**: 「私の意見を交えて」「あくまで私の考えです」などの断り
- **Zenn記法の使い方**: `:::message`、画像挿入、外部URLの貼り方
- **disclaimer**: 「ここで述べる内容は私個人の考えで、所属先の公式見解ではありません。」

### 2. ネタ・素材の収集

以下を確認して下書きの素材を集める：

1. **ユーザーからの直接入力**: チャットで伝えられた解釈・意見・不明点・書きたいこと
2. **_inbox/daily/**: 最新のトレンドファイルを確認
3. **_inbox/digest/**: 最新のURL要約ファイルを確認

不明な点があればユーザーに確認する：
- 「どのネタをメインにしますか？」
- 「あなたの独自解釈や意見はありますか？」
- 「読者ターゲットは技術者向け（tech）ですか、アイデア寄り（idea）ですか？」

### 3. 構成案の提示

下書きを書く前に、以下の構成案をユーザーに提示して確認を取る：

```
タイトル案: 「〇〇〇〇〇」
タイプ: tech / idea
想定読者: 〇〇
構成:
  ## はじめに
  ## 1. 〇〇とは
  ## 2. 〇〇の仕組み
  ## 3. 〇〇を使ってみた / 〇〇への私の解釈
  ## まとめ
```

### 4. 下書き生成

ユーザーの確認後、以下の著者文体ルールに従って下書きを生成する：

**必ず守るルール**:
- `## はじめに` から始める
- 難しい概念は必ず身近な例えで説明する
- 自分の意見・解釈を積極的に入れる（「〜だと思います」「個人的には〜」）
- 専門用語を初出時は必ず説明する
- 読者への配慮文を入れる（長い記事の場合はスキップ先を案内）
- 個人の見解であることの disclaimer を入れる（外部所属が関係するトピックの場合）
- 語尾は「〜です。」「〜ます。」基本だが、要所で「〜です！」「〜ですよね」を使う
- `:::message` はポイント・注意事項に使う
- 外部URLは本文中に直接貼る（Zennはリンクカードを自動生成する）
- 画像は `![](/images/filename.png =500x)` の形式（実際の画像がない場合は `<!-- TODO: 画像を追加 -->` とコメント）
- `## まとめ` で締める

**frontmatterテンプレート**:

```yaml
---
title: ""
emoji: ""
type: "tech"  # tech / idea
topics: []
published: false
---
```

### 5. ファイル保存

`_inbox/drafts/YYYYMMDD-{slug}.md` に保存する（YYYYMMDDは実行日）。

slugは記事内容を表す英数字とハイフンのみの文字列（例: `ai-security-supply-chain`）。

### 6. 完了報告

```
下書きを生成しました。

ファイル: _inbox/drafts/YYYYMMDD-{slug}.md
タイトル: {title}

確認して修正があればお知らせください。
問題なければ `/zenn-publish _inbox/drafts/YYYYMMDD-{slug}.md` で公開できます。
```

## 注意事項

- **既存記事を必ず読んでから**下書きを生成する（文体の一貫性が重要）
- ネタが複数ある場合は1記事1テーマに絞る
- 著者の実体験・意見が入らない「ただのまとめ記事」にならないよう注意する
- 不明な情報を憶測で書かない（`<!-- TODO: 確認が必要 -->` とコメントを入れる）
